# Non Generic Web Copy Guide

## Purpose

Replace vague AI-sounding marketing with copy that proves the site is real.

## Use When

- Any website build or polish pass with user-facing copy.
- Headlines, CTAs, forms, testimonials, pricing, service descriptions, case studies, or product sections feel generic.
- The user complains that the site is bland or template-like.

## Do Not Use When

- The user provided exact final copy and requested no copy changes.
- Legal/regulated copy where edits need explicit approval.

## Discovery Questions

- Who is being addressed and what do they need to do?
- What concrete product, service, place, person, or outcome is being described?
- What proof exists?
- What voice fits the category: technical, editorial, warm, restrained, direct, playful?

## Decision Tree

- If a line could fit any company, rewrite with a specific noun, audience, or proof.
- If copy is hype-heavy, replace claims with concrete evidence.
- If CTA is generic, name the actual action.
- If testimonials are fake/generic, use case notes or proof blocks instead.

## Implementation Rules

- Use concrete nouns and verbs.
- Make headlines specific to the offer or point of view.
- Keep subheads focused on audience, benefit, proof, or contrast.
- Use CTAs that match the page: view suites, book a tasting, ship a preview, see the work, compare plans.
- Microcopy should reduce uncertainty near forms and conversion points.

## Useful Patterns

- Product: Ship previews before your branch hits production.
- Service: Brand portraits for founders who hate looking staged.
- Luxury: Hand-finished leather goods made in small weekly runs.
- Booking: Reserve a 45-minute consultation.

## Anti-Patterns

- Unlock, elevate, seamless, revolutionary, cutting-edge, transform your experience.
- Learn more everywhere.
- Fake testimonial names and vague praise.
- Bullets where prose, a table, or a ledger would feel more credible.

## QA Checklist

- Check every headline and CTA for specificity.
- Remove leftover placeholder phrases.
- Check mobile line breaks and button labels.
- Ensure copy tone matches site category.

## Acceptance Criteria

- The copy cannot be pasted unchanged onto a random different site.
- The reader knows what is offered and what to do next.

## Shared Website Requirements

- Inspect before coding: project structure, framework, homepage/main entry, styling system, interaction/motion system, assets, and available commands.
- Before major changes, explain planned files and wait unless the user clearly says proceed, build, create, implement, fix, continue, or similar.
- Use existing project conventions first; explain major dependencies before adding them.
- Default to premium modern quality without generic SaaS/agency/template patterns.
- Prefer real, generated, or art-directed visuals when the subject benefits from images, video, product UI, or spatial media.
- Build version 1, inspect the rendered result, improve visual quality, then run available checks again.
- Check desktop and mobile. Keep text readable, avoid overlaps, respect focus states and reduced motion.
- Report files changed, commands run, visual checks completed, and anything not tested.
